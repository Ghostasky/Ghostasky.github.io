ASM:NtGlobalFlag

NtGlobalFlag字段默认为0，偏移：

  • x86: PEB+0x68
  • x64: PEB+0xBC

当调试器正在运行时, 该字段会被设置为一个特定的值.
尽管该值并不能十分可信地表明某个调试器真的有在运行, 但该字段常出于该目的而被使用
附加调试器不会更改 NtGlobalFlag 的值。但是，如果进程是由调试器创建的，则将设置以下标志：

  • FLG_HEAP_ENABLE_TAIL_CHECK (0x10)
  • FLG_HEAP_ENABLE_FREE_CHECK (0x20)
  • FLG_HEAP_VALIDATE_PARAMETERS (0x40)

ASM:HeapFlags

如果 HeapFlags 的值大于 2 说明程序处于调试状态

  • 在 32 位 Windows NT, Windows 2000 和 Windows XP 中, Flags位于堆的0x0C偏移处.
  • 在 32 位 Windows Vista 及更新的系统中, 它位于0x40偏移处
  • 在 64 位 Windows XP 中, Flags字段位于堆的0x14偏移处
  • 在 64 位 Windows Vista 及更新的系统中, 它则是位于0x70偏移处.
  • 在所有版本的 Windows 中, Flags字段的值正常情况都设为HEAP_GROWABLE(2)

ASM:HeapForceFlags

如果 HeapForceFlags 的值大于 0 则说明处于调试状态。

  • 在 32 位 Windows NT, Windows 2000 和 Windows XP 中, ForceFlags位于堆的0x10偏移处.
  • 在 32 位 Windows Vista 及更新的系统中, 它位于0x44偏移处.
  • 在 64 位 Windows XP 中, ForceFlags字段位于堆的0x18偏移处
  • 在 64 位 Windows Vista 及更新的系统中, 它则是位于0x74偏移处.

检查低碎片堆

当可执行文件在调试器下运行时，没有为其创建低碎片堆(LFH)，因此指向 LFH 的指针等于 NULL。
所以我们只需要检查指向 LFH 的指针是否为空，以检测进程是否在调试器中创建。

对于 WOW64 可执行文件，指向 LFH 的指针位于 Windows 7 下的“heap_handle +0xd4 ”
和64 位可执行文件的“heap_handle+ 0x178 ”。

  • Windows 7：
    • x86: heap_handle + 0xd4
    • x64: heap_handle + 0x178
  • win8 or win8.1
    • x86: heap_handle + 0xd0
    • x64: heap_handle + 0x170
  • win10 or later
    • 经常变，这里没写

API::NtQueryInformationProcess

除了调用 CheckRemoteDebuggerPresent 之外，还可以直接调用 NtQueryInformationProcess

ProcessDebugPort

ProcessDebugFlags

当使用ProcessDebugFlags类调用NtQueryProcessInformation函数时，该函数将返回EPROCESS->NoDebugInherit的反向值，这意味着如果存在调试器，则该函数将在进程被调试时返回FALSE。

ProcessDebugObject

使用反反调试技术来防止被反调试器绕过。

首先使用 NtQueryInformationProcess 函数查询当前进程是否存在调试器。如果查询的结果不是 STATUS_PORT_NOT_SET 或者返回的句柄不为空，则说明当前进程正在被调试，返回 TRUE。

如果第一次查询返回了 STATUS_PORT_NOT_SET，那么该函数将使用 NtQueryInformationProcess 函数进行第二次查询。这一次查询会在返回长度和调试对象句柄缓冲区重叠的情况下进行，以检测是否存在反反调试技术。如果查询的结果不是 STATUS_PORT_NOT_SET 或者返回的句柄为空，或者返回长度被错误地覆盖为非预期值，则说明当前进程正在被调试，返回 TRUE。

总之，这个函数通过两次查询来检测当前进程是否存在调试器，并使用反反调试技术来防止被反调试器绕过
